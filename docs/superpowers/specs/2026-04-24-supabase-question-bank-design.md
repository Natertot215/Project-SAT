# Supabase Question Bank ÔÇö Design

**Date:** 2026-04-24
**Status:** Approved (brainstorming)
**Scope:** Replace placeholder question content with a Supabase-hosted shared question bank. Wire the client to fetch questions before each session starts. Auth, accounts, and per-user history persistence remain on localStorage for v1.

---

## 1. Goals & Non-goals

### Goals

- Persist a shared bank of SAT practice questions in Supabase, queryable by section, skill, and difficulty.
- Replace the stubbed content layer (`getQuestionContent`, `generateCorrectAnswers`) with a single async fetch boundary in `src/lib/api.ts`.
- Pre-load every question for a session before the user sees question 1 ÔÇö no per-question network calls mid-session.
- Support a per-skill rolling cooldown so a user does not see the same question more than once within their last 200 questions of that skill.
- Surface a difficulty selector (`easy` / `med` / `hard` / `mixed`) in Practice and Test views.
- Gate the "Start" button on valid skill + question-count selection.

### Non-goals (deferred)

- Authentication / user accounts. Anonymous sessions only; cooldown lives in localStorage.
- Server-rendered API or edge functions. Direct browser-to-Supabase via anon key + RLS.
- Adaptive / dynamic difficulty within a session.
- Math rendering (KaTeX) ÔÇö the schema supports `$...$` delimiters in text but rendering is a separate task.
- Curation tooling (admin UI for adding / reviewing questions).
- Generation pipeline (Anthropic API ÔåÆ Supabase). The bank is populated out-of-band initially.

## 2. Schema

### Table `questions`

```
id              bigint generated by default as identity primary key
section         text not null check (section in ('rw','math'))
skill           text not null
difficulty      text not null check (difficulty in ('easy','med','hard'))
passage         text                       -- nullable, RW only
stem            text not null
choices         jsonb not null             -- ["A...", "B...", "C...", "D..."]
correct_index   int  not null check (correct_index between 0 and 3)
explanation     text not null
chart_data      jsonb                      -- nullable, structured chart spec
image_path      text                       -- nullable, Supabase Storage path
created_at      timestamptz default now()
```

**Indexes**
- `(section, skill, difficulty)` ÔÇö primary selection path
- `(skill)` ÔÇö for skill-only filters

**Why `bigint identity` and not `uuid`:** the existing `Question.id` and `HistoryEntry.id` types are `number`. Using bigint identity keeps JS-side numeric IDs and avoids a typewide refactor.

**Why no separate `passages` table:** real SAT paired-passage questions are uncommon enough that inline duplication is acceptable for v1. A future migration can normalize if needed.

**Why no `latex` flag column:** KaTeX is rendered from inline `$...$` delimiters in `stem` / `choices` / `explanation`. Intent is in the text; metadata column is over-engineered.

### Storage bucket `question-images`

- Public read.
- Written to by the bank-population pipeline only.
- `image_path` on a question row is the bucket-relative path.

### Row-level security (RLS)

- Enable RLS on `questions`.
- Policy: `SELECT` allowed for all (`using (true)`). No INSERT / UPDATE / DELETE from the client.
- `question-images` bucket: public read.

## 3. Client integration

### New file: `src/lib/supabase.ts`

A single client instance shared app-wide.

```ts
import { createClient } from "@supabase/supabase-js";

export const supabase = createClient(
  import.meta.env.VITE_SUPABASE_URL!,
  import.meta.env.VITE_SUPABASE_ANON_KEY!,
);
```

### Modified: `src/types.ts`

`Question` grows from a stub to the full row shape:

```ts
export type Difficulty = "easy" | "med" | "hard";
export type DifficultyChoice = Difficulty | "mixed";

export interface ChartSpec {
  // shape TBD when first chart question is authored
}

export interface Question {
  id: number;
  section: "rw" | "math";
  skill: Skill;
  difficulty: Difficulty;
  passage: string | null;
  stem: string;
  choices: [string, string, string, string];
  correctIndex: number;
  explanation: string;
  chartData: ChartSpec | null;
  imagePath: string | null;
}
```

`CorrectMap` and the `correctMap` field on `SessionState` are removed. Phases compute correctness from `answers[i] === questions[i].correctIndex`.

`SessionInit` fresh-start payload grows:

```ts
| { type: SessionType; n: number; skills: Skill[]; difficulty: DifficultyChoice }
```

A new `Phase` value is added: `"loading"`.

### Modified: `src/lib/api.ts`

Replaces both stubs with one async function:

```ts
export async function fetchQuestions(opts: {
  section: "rw" | "math";
  skills: Skill[];
  difficulty: DifficultyChoice;
  count: number;
  excludeIds: number[];   // per-skill cooldown buffer, deduplicated by caller
}): Promise<Question[]>
```

Behavior:

- Single difficulty: one query ÔÇö `select * from questions where section = $1 and skill = any($2) and difficulty = $3 and id <> all($4) order by random() limit $5`.
- Mixed: three queries in parallel (one per difficulty), each with the same filters, results concatenated **easy ÔåÆ med ÔåÆ hard**. Excluded IDs are extended after each completes so buckets don't collide.
- Maps snake_case rows ÔåÆ camelCase `Question` objects.
- If the result has fewer rows than `count`, retry once with empty `excludeIds` (relaxation step), then surface the deficiency to the caller as a thrown error if still short.

### Modified: `src/views/session/SessionView.tsx`

- `buildInitialState` becomes async only on the fresh-init path. Resume path stays synchronous.
- On mount, set `phase: 'loading'`, call `fetchQuestions` once for Practice or `Promise.all` of per-module calls for Test, then atomically swap to `phase: 'questions'`.
- Keep an abort ref so a Back-during-loading drops the resolved promise.

### Modified: `src/views/PracticeView.tsx` and `src/views/TestView.tsx`

- Add a difficulty selector button (matching the existing skill selector pattern) near the bottom of the page.
- Compute `canStart = skills.length > 0 && isValidCount(n)` and disable the Start button when false.

### New phase component: `src/views/session/LoadingPhase.tsx`

- Centered spinner using existing primitives.
- Label "Preparing your sessionÔÇª" ÔåÆ after 3s rotates to "Almost thereÔÇª".
- After 10s with no resolution, becomes error state with a Back button.
- Reuses session shell so the layout doesn't shift when the questions phase mounts.

### Removed / deprecated

- `getQuestionContent`
- `generateCorrectAnswers`
- `CorrectMap` type and `correctMap` on `SessionState`
- The `correctMap` validator branch in `storage.ts isValidSessionState` (replaced with validation of the new `Question` shape).

## 4. Selection algorithm

### Equal-split rule for `mixed`

```
base  = floor(count / 3)
extra = count % 3

easy = base + (extra >= 1 ? 1 : 0)
med  = base + (extra >= 2 ? 1 : 0)
hard = base
```

Remainder favors easier buckets first. Total always equals `count`.

### Mixed ordering

Concatenate buckets **easy ÔåÆ med ÔåÆ hard**. No within-bucket sort. Difficulty ramps monotonically across the session.

### Per-skill cooldown (no repeats within last 200 of a skill)

- Stored client-side in localStorage as `recentBySkill: Record<Skill, number[]>`.
- FIFO buffer per skill, capped at 200 IDs.
- Updated only on **session submit** (results phase), never mid-session ÔÇö abandoned sessions don't poison the buffer.
- Passed into `fetchQuestions` as `excludeIds` (union of buffers for the selected skills).
- Server query uses `id <> all($excludeIds::bigint[])`.
- One automatic relaxation: if the result is short, retry the same query with `excludeIds = []`. If still short, throw ÔÇö the loading phase shows the error.

### Validation gate

`canStart` on Practice / Test view:
- `skills.length > 0`
- `Number.isInteger(n) && n > 0 && n <= maxAllowedForView`
- `difficulty` is one of `'easy' | 'med' | 'hard' | 'mixed'`

The Start button is disabled (visually inert) when `canStart` is false. No toast, no submission.

### Future migration to accounts

Replace the localStorage buffer with `user_question_views(user_id, question_id, skill, viewed_at)` and rewrite the exclusion subquery server-side. The `api.ts` boundary absorbs the change; nothing upstream of `fetchQuestions` knows the storage swapped.

## 5. Pre-fetch contract

- `SessionView` enters `phase: 'loading'` immediately on mount for fresh sessions.
- A single fetch boundary (one query for Practice, parallel per-module queries for Test) runs to completion before transitioning to `phase: 'questions'`.
- The transition is atomic ÔÇö partial loads never reach the questions phase.
- Failure is terminal for session start, not for the running session: any module short of its requested count surfaces an error in the loading phase with a Back button.
- Resume path skips loading entirely because `SessionState` already contains all questions.

---

## 6. Performance at 8k rows

- `order by random() limit n` on filtered subsets of an 8k-row table is sub-50ms in Postgres.
- Indexes on `(section, skill, difficulty)` and `(skill)` cover every selection.
- The `id <> all(...)` exclusion array is bounded at 200 ÔÇö negligible cost.
- No `tablesample`, materialized views, or partitioning needed at this scale. Revisit at ~100k+ rows.
